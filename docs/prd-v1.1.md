# 家庭关怀小程序 V1.1 PRD

## 1. 文档信息

- 版本：V1.1
- 产品：家庭关怀小程序
- 目标读者：产品、前端、云函数开发、测试、运营
- 关联版本：V1.0（已上线基础能力：今日/健康/回忆/我的）

## 2. 背景与目标

### 2.1 背景

当前版本已具备家庭状态查看、报平安、健康摘要、相册、权限与隐私基础能力，但仍缺少以下高频价值场景：

- 关键时段未报平安的主动提醒
- 日常健康管理中的“用药/复诊”提醒
- 长辈在不便输入时快速发出求助
- 家庭成员对一周情况的低成本回顾
- Senior 模式下更极简、更大按钮的首页操作体验

### 2.2 V1.1 目标

1. 提升“报平安”触达与完成率，降低家属焦虑。
2. 增加健康类提醒闭环，提高日常使用频次。
3. 增强长辈端可用性与求助效率。
4. 形成周维度家庭关怀摘要，提升留存。

## 3. 范围边界

### 3.1 In Scope（本期纳入）

1. 异常未报平安提醒
2. 用药/复诊提醒
3. 一键求助卡
4. 周报自动生成
5. 长辈极简首页

### 3.2 Out Scope（本期不做）

- 家庭值班表（按需求排除）
- AI 健康诊断/治疗建议
- 对外社交广场、陌生人关系链
- 复杂多组织（一个账号加入多个家庭）治理

## 4. 角色定义

- admin：家庭管理员，管理成员与提醒策略
- adult：成年成员，可查看家庭数据并执行日常操作
- senior：长辈成员，使用极简交互

## 5. 功能模块 PRD

## 5.1 模块 A：异常未报平安提醒（P0）

### A1. 业务目标

- 在约定时间窗内，若成员未完成报平安，自动提醒指定家庭成员。
- 指标目标：
  - 报平安完成率提升 20%
  - 关键时段漏报率下降 30%

### A2. 用户故事

- 作为家属，我希望在父母未按时报平安时收到提醒，减少反复确认成本。
- 作为长辈，我希望提醒方式简单，不被频繁打扰。
- 作为管理员，我希望可配置提醒时段和提醒对象。

### A3. 入口与交互

- 入口：`今日 -> 通知策略（新增）`
- 配置项：
  - 监控时段（如 08:00-10:00）
  - 未报平安触发阈值（如 60 分钟）
  - 提醒对象（全体成人 / 指定成员）
  - 是否二次提醒（开关）

### A4. 规则

1. 成员在监控时段内完成报平安则不触发提醒。
2. 超过阈值仍未报平安，触发首次提醒。
3. 二次提醒开启时，首次提醒后 N 分钟仍未报平安再次触发。
4. 仅对 `adult/admin` 推送提醒，`senior` 默认不接收异常提醒。

### A5. 异常流程

- 网络不可用：记录待触发任务，恢复后补发一次提醒。
- 成员当天请假/静默：该时段不触发提醒。
- 重复触发保护：同成员同时间窗最多 2 次。

### A6. 数据模型

- 集合：`checkin_policies`
  - `_id`
  - `householdId`
  - `enabled` (bool)
  - `start_time` (HH:mm)
  - `end_time` (HH:mm)
  - `threshold_minutes` (number)
  - `second_reminder_enabled` (bool)
  - `second_reminder_minutes` (number)
  - `target_openids` (array)
  - `updated_by`
  - `updated_at`
- 索引建议：`householdId + enabled`

### A7. 埋点

- `checkin_policy_updated`
- `checkin_missed_alert_triggered`
- `checkin_completed_after_alert`

### A8. 验收标准

1. 未报平安超阈值时触发提醒记录。
2. 同时段重复触发次数不超过规则上限。
3. 关闭策略后不再触发提醒。

## 5.2 模块 B：用药/复诊提醒（P0）

### B1. 业务目标

- 建立家庭健康提醒机制，提高按时用药与复诊执行率。

### B2. 用户故事

- 作为家属，我希望设置父母每日用药提醒。
- 作为长辈，我希望收到简单明确的提醒，不需要复杂操作。
- 作为管理员，我希望可管理全家提醒计划。

### B3. 入口与交互

- 入口：`健康 -> 用药/复诊提醒（新增卡片）`
- 操作：
  - 新建提醒（类型：用药/复诊）
  - 设置时间、重复规则、备注
  - 标记“已完成”

### B4. 规则

1. 用药提醒支持每日/每周重复。
2. 复诊提醒支持单次与提前 N 天提醒。
3. 当天提醒完成后状态变更为“已完成”，不再重复弹出。
4. 支持成员维度可见性（仅本人/家庭可见）。

### B5. 异常流程

- 提醒时间过去后补登：允许补打卡并记录时间偏差。
- 重复提醒冲突：同分钟多条提醒合并展示。

### B6. 数据模型

- 集合：`care_reminders`
  - `_id`
  - `householdId`
  - `owner_openid`
  - `type` (`medicine` / `followup`)
  - `title`
  - `note`
  - `remind_at`
  - `repeat_rule` (`none`/`daily`/`weekly`)
  - `advance_days` (number, followup)
  - `status` (`active`/`done`/`archived`)
  - `visibility` (`self`/`household`)
  - `created_at` / `updated_at`
- 索引建议：`householdId + owner_openid + status + remind_at`

### B7. 埋点

- `reminder_created`
- `reminder_done`
- `reminder_overdue`

### B8. 验收标准

1. 可创建、编辑、删除提醒。
2. 到点可触发提醒，完成后状态正确更新。
3. 复诊提前提醒逻辑正确执行。

## 5.3 模块 C：一键求助卡（P1）

### C1. 业务目标

- 在弱输入场景下，让长辈 1-2 步完成“需要帮助”通知。

### C2. 用户故事

- 作为长辈，我希望点一个按钮就通知家人联系我。
- 作为家属，我希望知道求助类型和时间，便于快速响应。

### C3. 入口与交互

- 入口：`今日 -> 一键求助（新增卡片）`
- 按钮：
  - `请联系我`
  - `需要陪同`
  - `身体不适`
- 提交后展示“已发送给家人”状态与撤回入口（短时可撤回）。

### C4. 规则

1. 每 5 分钟内相同求助类型防抖（避免误触多发）。
2. 可选语音补充（后续增强，可先不做）。
3. 求助默认通知家庭 `adult/admin` 成员。

### C5. 数据模型

- 集合：`help_requests`
  - `_id`
  - `householdId`
  - `sender_openid`
  - `type` (`call_me`/`companionship`/`unwell`)
  - `message`
  - `status` (`sent`/`acknowledged`/`cancelled`)
  - `created_at`
  - `ack_at`
- 索引建议：`householdId + created_at(desc)`

### C6. 埋点

- `help_request_sent`
- `help_request_acknowledged`
- `help_request_cancelled`

### C7. 验收标准

1. 长辈可一键发送求助并看到发送成功。
2. 防抖规则生效，不会短时间重复刷屏。
3. 家属侧可查看求助记录与时间。

## 5.4 模块 D：周报自动生成（P1）

### D1. 业务目标

- 自动汇总家庭一周状态，减少手工整理成本，提升回访频次。

### D2. 用户故事

- 作为家属，我希望每周看到一份摘要，快速了解父母状态趋势。
- 作为管理员，我希望统一查看关键指标（报平安、健康、互动）。

### D3. 入口与交互

- 入口：
  - `今日` 顶部“本周摘要”
  - `我的` -> “家庭周报”
- 展示内容：
  - 本周报平安次数
  - 健康摘要（步数/睡眠均值）
  - 相册新增数量
  - 重点事件（求助/复诊）

### D4. 规则

1. 每周日 22:00 生成本周周报（可手动补生成）。
2. 数据缺失字段显示“暂无数据”，不阻断生成。
3. 仅家庭成员可查看所属家庭周报。

### D5. 数据模型

- 集合：`weekly_reports`
  - `_id`
  - `householdId`
  - `week_start`
  - `week_end`
  - `checkin_count`
  - `health_summary` (object)
  - `album_new_count`
  - `highlights` (array)
  - `generated_at`
- 索引建议：`householdId + week_start(desc)`

### D6. 埋点

- `weekly_report_generated`
- `weekly_report_viewed`
- `weekly_report_shared`（若后续支持分享）

### D7. 验收标准

1. 自动生成任务按周产出记录。
2. 页面可稳定读取最近周报。
3. 无数据周仍能生成空模板周报。

## 5.5 模块 E：长辈极简首页（P0）

### E1. 业务目标

- 降低长辈用户操作门槛，提高关键动作完成率。

### E2. 用户故事

- 作为长辈，我希望首页只有最常用功能，字大、按钮大、步骤少。
- 作为家属，我希望长辈误触少、关键操作更稳定。

### E3. 入口与交互

- 触发条件：`Senior 模式开启` 或 角色为 `senior`
- 首页改造：
  - 保留 3 个主按钮：`报平安`、`一键求助`、`看家人动态`
  - 辅助入口折叠到“更多”
  - 字号与按钮热区提升

### E4. 规则

1. Senior 模式下优先展示极简首页，不破坏原有导航。
2. 未开启 Senior 模式保持标准首页。
3. 可随时切回标准模式。

### E5. 数据模型

- 使用现有设置项扩展：`members.is_senior_mode`
- 无新增核心集合，仅新增 UI 状态字段与埋点

### E6. 埋点

- `senior_mode_enabled`
- `senior_home_action_click`
- `senior_home_exit`

### E7. 验收标准

1. Senior 模式开关即时生效。
2. 极简首页核心操作 2 步内可完成。
3. 关闭后回到标准版首页且数据不丢失。

## 6. 数据与权限策略（跨模块）

1. 所有业务数据需带 `householdId` 作为租户隔离键。
2. 云函数写入时基于 `openid` 校验成员身份。
3. admin 才可修改策略类配置（提醒策略、成员角色）。
4. 上传文件路径需按 `householdId` 进行目录隔离。

## 7. 指标体系（V1.1）

- 核心行为：
  - 日报平安完成率
  - 异常提醒触达率/转化率
  - 用药提醒完成率
  - 求助卡触发后响应时长
  - 周报查看率
- 产品健康：
  - D1/D7 留存
  - Senior 用户任务完成率

## 8. 发布与排期建议

### 8.1 优先级

- P0：异常未报平安提醒、用药/复诊提醒、长辈极简首页
- P1：一键求助卡、周报自动生成

### 8.2 里程碑（建议）

- M1（1 周）：完成 P0 需求开发 + 联调 + 冒烟
- M2（第 2 周）：完成 P1 需求开发 + 回归
- M3（第 3 周）：灰度发布 + 指标观测 + 问题修复

## 9. 风险与应对

1. 风险：提醒能力依赖系统通知权限  
   应对：在关键节点提示开启通知，并提供失败兜底提示。

2. 风险：CloudBase 集合权限配置错误导致越权  
   应对：上线前进行越权测试与规则审计。

3. 风险：长辈误触高频求助  
   应对：求助按钮防抖 + 二次确认开关（可配置）。

4. 风险：周报自动任务遗漏  
   应对：提供手动重生成入口与失败告警。

## 10. 测试验收清单（对应模块）

- 异常提醒触发/取消/二次提醒规则通过
- 用药/复诊创建、提醒、完成全链路通过
- 一键求助发送、防抖、家属侧可见通过
- 周报自动生成、读取、空数据兜底通过
- Senior 首页切换与关键动作可达性通过
